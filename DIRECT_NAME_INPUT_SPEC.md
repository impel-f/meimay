# 候補名の直接追加 仕様案

## 目的

ユーザーがすでに候補として考えている名前を、探索フローを経由せずにアプリへ記録できるようにする。

この機能は「名前候補を保存済みに追加する」導線であり、読み探し・漢字探し・ビルドの代替ではなく、持ち込み候補を管理対象に入れるための入口として扱う。

## 基本方針

- 無料会員でも候補名の直接追加はできる。
- 入力後は確認プレビュー画面を挟まず、保存済み画面へ遷移する。
- 直接追加した名前は `保存済み` に入れる。
- 直接追加した名前は読みストックへ反映し、漢字は利用権付きでFreeストックへ反映する。
- 人名用漢字を含む名前でも、名前表記そのものは無料会員にも表示する。
- 人名用漢字の詳細・意味・AI深掘り・素材としての再利用は、現在の会員状態と既存ストック有無で判定する。

## 表示文言

入口の文言は `候補名を追加` を基本にする。

画面内の主ボタンは `保存済みに追加` とする。

`保存` だけだと、ビルド済み候補の保存と区別しにくいため使わない。

## 入口

### 1. 初回ウィザード

最初の進め方選択に、候補名を直接入れる導線を追加する。

選択肢:

- `候補名を直接指定`
- 説明: `名前と読みを入力して保存済みに入れます`

初回ユーザーが「もう名前案はある」状態で迷わないため、ここに置く。

### 2. 探す

探すの `進め方を選ぶ` 画面では、ウィザードと同じトンマナのカードとして入口を置く。

想定配置:

- 読み候補を探す
- 漢字を探す
- 候補名を直接指定

探す画面のホームにも、軽い補助導線として `候補名を追加` を置く。

保存済み画面だけに置くと、初回ユーザーが発見しにくいため、探す導線にも置く。

### 3. 保存済み画面

保存済み画面内の追加導線は、2026-07-11 時点では一旦非表示にする。

入口が増えすぎると、保存済み画面の主目的である「候補の確認」と競合するため、探す画面の導線に寄せる。

## 入力画面

入力項目:

- 名前
  - 例: `陽斗`
  - 必須
- 読み
  - 例: `はると`
  - 必須
- メモ
  - 任意

バリデーション:

- 名前は漢字・かな・記号を許容するが、保存名として空は不可。
- 読みはひらがなで入力し、カタカナはひらがなへ正規化して保存する。漢字・英字・記号を含む読みは保存しない。
- 名前と読みの文字数は既存の保存済み表示で崩れない範囲に制限する。
- 名前に含まれる漢字は、常用漢字・人名用漢字を含む漢字マスタに存在するものだけ許可する。
- 漢字マスタに存在しない漢字が含まれる場合はアラートを出し、保存しない。
- ひらがな・カタカナは保存できる。
- 漢字以外の文字は、漢字マスタ存在チェックの対象にしない。
- 記号・空白は保存名として破綻しない範囲で許可する。
- `々` など、名前で使われる反復記号は漢字マスタチェック対象外として許可する。

未登録漢字のアラート文言:

`「{文字}」はこのアプリの漢字データにありません。登録済みの常用漢字・人名用漢字だけ保存できます。`

未登録漢字の追加リクエスト導線は作らない。

未登録漢字は制度上またはアプリの対応範囲外として扱い、保存不可にする。

確認プレビュー画面は作らない。

入力後はそのまま保存し、保存済み画面に遷移する。

保存後に、読み・漢字ストックへ入ったことを説明する追加トーストや完了メッセージは出さない。

直接追加の主目的は保存済みに入れることであり、ストック同期は内部的な補助として扱う。

## 保存データ

直接追加した保存済みには、通常の保存名に加えて出自を持たせる。

```js
{
  source: "direct-name",
  givenName: "陽斗",
  givenReading: "はると",
  message: "",
  createdAt: "...",
  updatedAt: "...",
  directInput: true
}
```

漢字ごとの詳細が作れる場合は、既存の保存済み詳細表示で扱えるように `combination` 相当も保持する。ただし、直接入力だけでは漢字ごとの読み区切りを確定できないため、各文字へ `sessionReading` や読みセグメントを割り当てない。

マスタに存在しない漢字は保存しない。

## ストック反映

### 無料会員

- 保存済みに追加する。
- 読みストックには「漢字を選んでいない読み」として追加する。
- 常用漢字は読みと紐づけず、Freeの漢字ストックに追加してよい。
- 人名用漢字はFreeの漢字ストックへ鍵付きで追加する。
- 鍵付き人名用漢字は保存名の中では表示するが、詳細・AI深掘り・ビルドでの再利用はロックする。
- 鍵付きの段階では漢字ランキングの票に加えない。

### プレミアム会員

- 保存済みに追加する。
- 読みストックへ「漢字を選んでいない読み」として追加する。
- 常用漢字・人名用漢字を含めて、読みと紐づけずFreeの漢字ストックへ追加する。
- 既存ストックに同じ読み/漢字がある場合は重複追加しない。

直接指定した名前は漢字と読みの正確な区切りを特定できないため、入力順をそのまま読みスロットへ割り当てない。読みストックから漢字選択へ進む場合は、漢字未選択の状態から開始する。

### 有料から無料に戻ったユーザー

保存済みの名前はそのまま残す。

現在無料でも、過去にストックへ入っている人名用漢字は情報を見られる扱いにする。無料の直接指定で鍵付きになった漢字も、購入・無料体験・パートナー特典のいずれかでプレミアムが有効になった時点で永久解放する。

理由:

- 有料時に自分で選んで保存した候補を、無料化後に急に読めなくする体験は悪い。
- 保存済みやストックはユーザーの作業成果なので、あとから不可視にしない方がよい。
- プレミアムの価値は「新しく人名用漢字を探す・追加する・AIで広げる」部分に置く。

現在無料の場合にロックするもの:

- ストックにない人名用漢字の意味詳細
- ストックにない人名用漢字のAI漢字深掘り
- ストックにない人名用漢字を素材として再利用する操作
- 人名用漢字を新たにストックへ追加する操作

過去にプレミアム時代にストックへ入れた人名用漢字は、無料化後も詳細を閲覧できる。

永久解放済みの漢字を再び直接指定しても、鍵付きへ戻したり重複追加したりしない。

## 名前詳細でのロック表示

保存済み詳細では、名前表記と読みは常に表示する。

無料会員で人名用漢字を含む場合:

- 対象文字が既存ストックにある場合は、通常通り詳細を表示する。
- 対象文字が既存ストックにない場合は、`人名用👑` または鍵表示を付ける。
- ストックにない人名用漢字の意味・由来・AI深掘りなどの詳細領域はロック表示にする。
- ストックにない人名用漢字の `この漢字を候補に追加` のような再利用ボタンはロックする。

無料会員でも「その名前を保存済みとして眺める」ことは妨げない。

## 既存機能との関係

### フリーストック

直接追加は、フリーストックそのものではない。

保存済みに追加する導線であり、漢字ストックへの反映は会員状態に応じた副作用として扱う。

直接追加した漢字は `sessionReading: "FREE"` として保持する。無料会員の人名用漢字には `stockAccess: "premium-required"` を付け、表示はするが詳細・ビルド利用は共通判定で遮断する。

プレミアム有効時は `stockAccess: "unlocked"` へ昇格し、期限終了後も戻さない。属性が存在しない既存ストックは後方互換のため解放済みとして扱う。

### ビルド

直接追加は、ビルド結果の保存とは別扱いにする。

ビルド画面の中心導線にはしない。

### 漢字検索

漢字検索内の小カードに直接追加入口を置くのは避ける。

漢字検索は「漢字を探す」文脈なので、名前全体を入力する導線を混ぜると目的がぼやける。

## 連携・同期

直接追加した保存済みは、通常の保存済みと同様にパートナー連携・子ども別ワークスペースへ同期する。

ただし、相手側での人名用漢字の詳細閲覧は、相手側または連携特典を含む現在の会員状態で判定する。

## 分析イベント

追加したいイベント:

- `direct_name_input_opened`
  - source: `wizard` / `search_footer` / `saved_empty` / `saved_header`
- `direct_name_input_saved`
  - has_jinmei: 0/1
  - kanji_count
  - reading_length
  - source
- `direct_name_input_stock_synced`
  - reading_synced: 0/1
  - kanji_synced_count
  - locked_jinmei_count
  - premium: 0/1

## 実装メモ

1. `保存済み` に直接追加用の作成関数を用意する。
2. 入力画面を追加する。
3. ウィザード・探すフッター・保存済みに入口を追加する。
4. 保存時に会員状態を見て、読み/漢字ストック同期を行う。
5. 保存済み詳細で、ストックにない人名用漢字の詳細ロックを表示する。
6. 連携・バックアップ・子ども別ワークスペースの正規化で `source: "direct-name"` を保持する。

## 現状メモ

2026-07-05 時点の実装では、会員判定は主に入口側で行われている。

- `isKanjiAccessibleForCurrentMembership(item)` は、プレミアムなら全漢字、無料なら `常用漢字` のみ許可する。
- 漢字検索・読み候補・AI提案など、新しく候補へ追加する入口では人名用漢字がロックされる。
- 一方で、漢字詳細モーダル本体は会員状態による表示ロックを持っていない。
- 保存済み詳細から漢字をタップした場合は、マスタから補完して `showKanjiDetail(data)` を直接開くため、保存済み経由では人名用漢字の詳細が見える可能性がある。

この現状は、「過去に保存・ストックした人名用漢字は無料化後も情報を見られる」という新仕様に近い。

実装時は、以下のように整理する。

- 新規追加・新規探索は現在の会員状態でロックする。
- 既存ストックにある人名用漢字は、無料化後も詳細閲覧を許可する。
- 保存済みに含まれるだけでストックにない人名用漢字は、無料では詳細をロックする。
