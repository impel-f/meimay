# Meimay 公開までの実行計画

更新日: 2026-03-08
目標: 2026年3月31日までに iOS / Android を申請可能な状態にする

## 1. 現状サマリー

Meimay はすでに以下の土台ができています。

- 命名フローの中核
  - 読みから漢字を探す
  - 響きから探す
  - 直感スワイプ
  - ビルド
  - 姓名判断
- 周辺機能
  - 漢字検索
  - ランキング
  - 今日の一字
  - 保存履歴
  - ペアリング
- 将来の拡張を見据えた要素
  - Firebase 匿名認証
  - ルームコード連携
  - AdMob 前提の設計
  - プレミアム画面のたたき台
  - 利用規約 / プライバシーポリシーのアプリ内表示

一方で、今のままでは「Webアプリとしてかなり進んでいる」段階であり、App Store / Google Play にそのまま出せる状態ではありません。

## 2. 最優先の公開ブロッカー

### A. ネイティブ配布の器がない

現状は `public/index.html` + Vanilla JS の Web/PWA 構成で、`android/` `ios/` `capacitor.config.*` がありません。

今月中に Apple / Google へ出すなら、まず以下のどちらかが必要です。

- 推奨: Capacitor で iOS / Android シェル化
- 代替: Expo / React Native へ載せ替え

今の構成を活かすなら、今月中の現実解は Capacitor です。

### B. 課金が実課金になっていない

現状のプレミアムは `localStorage` の ON/OFF だけで、StoreKit / Google Play Billing の実装がありません。

- Apple はアプリ内のデジタル機能解放に In-App Purchase を要求
- Google Play もデジタル商品は Play Billing 前提

したがって、今の「プレミアム有効化」ボタンは申請前に次のどちらかにする必要があります。

- 本命: 本物の課金に置き換える
- 時短案: 申請版では課金導線を外し、無料 + 広告のみで先に出す

### C. ストア公開用の法務 / サポート URL が不足

アプリ内法務文言はありますが、ストア審査向けには不足があります。

- 公開 URL のプライバシーポリシー
- 公開 URL のサポートページ
- 開発者名 / 連絡先
- データ保持 / 削除ポリシーの明確化
- Firebase / Gemini / 広告 SDK を含めたデータ取り扱いの棚卸し

### D. ダッシュボードが未完成

ホーム上部の分析カードは現状「準備中アラート」です。

申請時に目立つ導線が未実装のままだと、完成度を落とします。ホームは「毎日開きたくなる理由」を作る中核なので、ここは優先して埋めるべきです。

### E. レイアウト崩れの火種がある

画面全体に `100vh` / `overflow: hidden` / ネストした `h-screen` が混在しており、端切れ・下部欠け・法務画面やモーダルの見切れが起こりやすい構成です。

とくに以下は要注意です。

- body 固定高さ
- screen 絶対配置 + 大きい bottom padding
- 一部画面の `h-[calc(100vh-180px)]`
- bottom nav / 広告 / safe-area の重なり

## 3. 今月中にやるべき順番

### Phase 1: 3/8 - 3/12 公開基盤を固める

#### 1. 配布方式を決める

推奨方針:

- 今月公開優先なら Capacitor を採用
- Web資産を最大限流用
- `ios/` `android/` を作成
- bundle id / package name / app name を固定

#### 2. 申請版スコープを決める

おすすめ:

- v1 公開必須
  - 読み検索
  - 直感スワイプ
  - 漢字検索
  - ビルド
  - 姓名判断
  - 保存
  - ペアリング基本
- v1 で仕上げる
  - ホームダッシュボード
  - ペア後の一致可視化
  - レイアウト崩れ
  - 法務 / サポート / ストア素材
- v1.1 に逃がせる
  - コメント共有
  - エクスポート
  - 高度な分析
  - プレミアムの多段プラン化

#### 3. まず消すべき審査リスク

- 「準備中」「未完成」文言の除去
- ダミー課金導線の整理
- 壊れた導線 / 空ページ / 死んだボタンの除去
- 表示だけある未連動UIの整理

### Phase 2: 3/13 - 3/18 体験の芯を磨く

#### 4. ホームダッシュボードを完成させる

目標は「次に何をすればいいかが一目でわかる」ことです。

実装したい内容:

- いまの進捗
  - 保存した漢字数
  - 保存した読み数
  - 保存名数
- 今日やること
  - 今日の一字
  - 続きから再開
  - パートナーと一致した候補
- 好みの見える化
  - 上位タグ 3つ
  - 好みの系統
  - 最近よく選ぶ画数帯 / 雰囲気

参考にしたい体験:

- BabyCenter 型: 毎日ひらく理由が明確
- Tinder 型: 迷わず一歩進める
- Peanut 型: 人とのつながりや安心感が見える

#### 5. ペアリング後の挙動を強化する

現状は「相手データを取り込む」寄りで、二人で決める体験の見せ方が弱いです。

今月中に最低限入れたいもの:

- 両者 Like の一致表示
- 一致候補の専用タブ or セクション
- パートナー由来の候補に role 表示
- 保存名の一致 / 片想い の区別
- 一致したときの小さな演出

今月中に入れられるなら強いもの:

- 候補へのひとことメモ
- 「なぜ好きか」タグ
- ペアでの最終候補ピン留め

#### 6. 漢字マッチングを拡充する

すでに一部の読みマッチングはかなり工夫されています。ここは「より親が安心して使える精度」に寄せます。

優先度順:

- 促音 / 清音 / 名乗りの扱いの再検証
- ぶった切り読みの説明表示
- 常用漢字 / 人名漢字 / 全件 の切替
- 読みに対する「おすすめ理由」の表示
- 検索結果のノイズ削減

### Phase 3: 3/19 - 3/24 マネタイズと法務を完成させる

#### 7. プレミアム方針を決める

結論から言うと、Meimay の強みは「親が迷いながら二人で絞れること」なので、コア価値は無料で触らせるべきです。

おすすめの無料 / 有料の切り分け:

無料で残す:

- 基本の命名フロー
- 漢字検索
- 保存
- ペアリング基本
- 今日の一字

プレミアムで強くする:

- 広告非表示
- 1日の直感スワイプ上限解除
- 全2999字の開放
  - 無料は「おすすめ + 常用 / 人名の定番優先」
  - 有料は全解放
- AI由来の回数上限緩和
- 一致分析ダッシュボード
- ペアメモ / 最終候補管理
- 名付けの軌跡エクスポート

おすすめ商品設計:

- 月額プラン
- 買い切りプラン

今月公開を優先するなら:

- 3/20 までに Billing 実装の目処が立たない場合は v1 は無料 + 広告で出す
- 課金は v1.1 で入れる

#### 8. 法務ページをストア提出レベルにする

最低限必要:

- Privacy Policy の公開ページ
- Terms の公開ページ
- Support ページ
- 問い合わせ先メール
- データ削除方法
- データ保持期間
- 第三者サービス一覧
  - Firebase
  - Firestore
  - Gemini
  - AdMob / AdSense

注意:

- 「個人情報は一切収集しません」は表現を再検討
- 匿名ID、利用ログ、広告SDK、Firebase 利用は整理して明示する
- ストアの Data safety / App privacy と本文を一致させる

### Phase 4: 3/25 - 3/31 申請と最終品質

#### 9. ストア素材を完成させる

## 4. デザイン / UI 改善の方向性

### 改善テーマ

- 安心感
- 決めやすさ
- 二人で進んでいる感
- 毎日開きたくなる軽さ

### すぐ効く改善

- ホームを「メニュー」ではなく「次の一歩」中心にする
- 直感スワイプにご褒美感を足す
- 一致候補に明確な勝ち筋を作る
- 名前候補の比較をしやすくする
- 画面ごとの主ボタンを1つに絞る

### 画面別

ホーム:

- 今やることを1つだけ強調
- 続きから再開カード
- 今日の一字
- パートナー一致カード

スワイプ:

- いま何を選んでいるかを常に上部表示
- Like / Nope / Super の意味を小さく固定表示
- 10枚ごとの区切りに達成演出

ビルド:

- 候補比較をもっと見やすく
- 画数 / 読み / タグ / 相手評価を1カードに統合

保存済み:

- 第一候補 / 保留 / 却下前 の3分類
- パートナー一致順ソート

## 5. コード上の要注意ポイント

公開前に必ず手を入れたいもの:

- ホームの分析カードが未完成
- 日次上限ロジックはあるが UI 未接続、実運用にも未使用
- プレミアム状態が実課金連動していない
- ペアリング後は「取り込み」中心で、相互一致の主役化ができていない
- PWA を名乗っているが Service Worker が見当たらない
- 広告が AdSense と AdMob の二重系統になっていて整理が必要
- Firebase 利用前提なのに Firestore ルール運用確認項目が見えない

## 6. 今月の現実的な勝ち筋

最短で勝つなら次です。

1. まずネイティブ化する
2. ホームダッシュボードを完成させる

- 迷わない
- 気持ちいい
- 二人で決めやすい

辞書の件数やAIよりも、最初の10分でこの3つを感じられることを優先するのが正解です。
