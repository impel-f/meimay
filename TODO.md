# Meimay プロジェクト ToDo ＆ ロードマップ整理

## 1. トップUIの刷新（フェーズ・文脈依存型）
現在のメニュー型UIから、ユーザーの「フェーズ」に合わせて次の一歩を提案するトップ画面へ改修する。

- [ ] **フェーズ判定ロジックの実装（ハイブリッド型）**
  - 初回起動時・オンボーディング時のウィザード（「読みは決まっていますか？」等を軽くヒアリング）
  - 行動ログに基づく自動判定（特定の読みにLikeが集中した場合や保存アクションが続いた場合にフェーズ移行を提案）
- [ ] **フェーズ別「今日の◯◯」UIの出し分け実装**
  - **未決定フェーズ:** 「今日の響き」 （ランダム・人気な読みの提案）
  - **読み決定フェーズ:** 「今日の漢字」 （決まった読みに合致する、様々な意味の漢字の提案）
  - **収束フェーズ:** 「今日の振り返り」 （保存済み候補、パートナーと一致した候補の再提示）

## 2. 共同命名体験（パートナー連携）の強化
Meimayのコアバリュー・MOATとなる「二人で決める体験」を軸にした機能追加。

- [ ] **ペア連携の強化・同期**
  - 招待リンク / QRコード / ルームコードでのシームレスなペアリング機能のブラッシュアップ
- [ ] **評価一致の可視化の実装**
  - 双方が「Like」した候補の強調表示（エフェクト、専用バッジなど「嬉しい体験」の演出）
  - 片方のみの「Like」との視覚的な差別化
- [ ] **コメント・メモ機能**
  - 候補（名前・漢字）ごとにテキストメモを残す機能
  - 「家族3人で呼びやすい」など、思考プロセスを記録しパートナーとリアルタイム共有
- [ ] **選好プロファイルの分析・比較**
  - Likeしたデータから「爽やか」「古風」などの傾向を分析・可視化
  - パートナーとの好みの傾向の違いを比較できるUI

## 3. 「育つ・貯まる」体験の実装（継続ログイン動機）
- [ ] **命名プロフィールの育成**
  - これまでに出会った（スワイプした）読みの数、保存した数の統計可視化
- [ ] **マイルストーン連動**
  - 出産予定日を設定し、「予定日まであと◯日。そろそろ読みを絞り込みませんか？」などのカウントダウン連動UI

## 4. マネタイズ・長期価値（将来の拡張）
- [ ] **「名付けの軌跡」のエクスポート / 冊子化の要件定義**
  - 悩んだ候補、お互いのメモ、最終決定に至るストーリーを抽出できるデータ構造の維持

## 💡 Antigravity (Gemini) からの意見・提案

ご提示いただいた機能要件とプロダクト思想は、ただの「辞書アプリ」から**「共同意思決定プラットフォーム」**へと引き上げる非常に素晴らしいアプローチだと感じます。特にパートナー間の「プロセスの共有」は体験価値が極めて高いです。

**1. 実装の優先順位について**
まずは最も価値を生み出し、Meimayの強みとなる **「2. 共同命名体験（特に『Likeの一致可視化』と『メモの共有』）」** から手を付けるのがおすすめです。ここのデータ構造とUIが固まっていれば、トップ画面（収束フェーズの今日の◯◯等）も作りやすくなります。

**2. 音のシミュレーション（手軽なMVPとして）**
「声に出した時の響き」を重視するなら、本格的なAI音声合成を導入する前に、まずはブラウザ標準の `Web Speech API` を使えば**数行のコードでプロトタイプ実装が可能**です。「スピーカーアイコンを押すとフルネームが再生される」という体験がどれくらい刺さるか、即座に検証できます。

**3. ステージ判定の「ウィザード」はライトに**
毎回聞かれると重いという課題に対しては、初回ウィザードで「まだ決まっていない」「読みは決まっている」等の大きな分岐だけ聞き、あとは「スキップしてはじめる」を押しやすくしておくのが良いでしょう。その後は裏側の「行動ログ（スワイプ傾向や検索履歴）」で自動追従するハイブリッド型が最もUXとして優れています。

**4. 姓名判断の扱い（外部APIの活用）**
姓名判断ロジックをすべて自前で持つのは大変です。あくまで決断の「背中を押す」要素として扱うなら、収束フェーズ用の専用画面でだけ外部API等を叩き、「大吉！」などのポジティブな結果や総画数だけをシンプルに表示するのが、実装コストとUXのバランスが良いと思います。
